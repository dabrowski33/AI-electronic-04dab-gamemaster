# GenRL GameMaster AI
## Comprehensive Project Documentation: PRD, ARD & SDLC PLAN

This technical documentation constitutes the formal functional and architectural specification for the advanced neuroevolutionary system **GenRL GameMaster AI**, designed for automated learning and strategy optimization in two-dimensional game environments (2D Simulators such as *Super Mario* and *Pacman*). The system integrates Reinforcement Learning (RL) techniques with meta-optimization driven by Genetic Algorithms (GA), coordinated via a multi-agent LLM/LRM orchestration framework.

---

## 1. PRD (Product Requirements Document)

### 1.1. Goal and Product Vision
The primary objective of **GenRL GameMaster AI** is to build a self-learning agent that masters a 2D game environment from scratch, without any prior knowledge of game rules, collision maps, or scoring logic. It achieves this solely based on terminal game states (win/loss) and sparse reward signals. The core innovation lies in embedding a Genetic Algorithm loop supervised by Large Reasoning Models (LRMs). These models analyze the evolutionary trajectory of code and neural weights, dynamically mutating the algorithm's code structure and hyperparameters to break out of local performance minima.

### 1.2. Target Audience (Personas)
* **AI Researcher / R&D Engineer (The Researcher):** An expert seeking a flexible environment to experiment with hybrid evolutionary-reinforcement learning algorithms. Requires absolute visibility into mutation logs, genome population structures, and fitness convergence graphs.
* **QA Engineer / Automation Tester (The Verifier):** A specialist utilizing the autonomous agent to detect map flaws, invisible walls, or glitches in newly designed 2D games through mass, non-human exploration of levels.

### 1.3. Key Features and Product Requirements
1. **Automated Environment & State Parser:** A module that reads raw pixel matrices or feature vectors from 2D simulators (*Super Mario* / *Pacman*) and maps them directly into an RL state space.
2. **Evolutionary Orchestrator (LangGraph Loop):** A stateful graph governing the lifecycle of the agent population. It handles generation evaluation, selection, algorithmic code mutation, and parameter crossover.
3. **Isolated Execution Worker (OpenCode Sandbox):** A secure runtime component that executes mutated Python/C++ code responsible for the agent's interaction with the game.
4. **Cognitive Cascading Engine (Hermes 3 / Codex):** An intelligent layer that generates reward shaping modifications and network structure mutations using the Hermes 3 model via OpenRouter, with a redundant failover switch to OpenAI Codex.
5. **Institutional Control Dashboard (NBP-Style UI):** A visual presentation layer built on rigorous financial design aesthetics, eliminating distractions to focus purely on core convergence metrics and telemetry.

### 1.4. Specific User Stories

> **US-01: Initialization of Autonomous Training in a 2D Environment**
> * **As an** AI Researcher,
> * **I want to** select a simulator type (e.g., Pacman) from the UI and define a base objective function,
> * **So that** the system launches the first generation of agents to learn the game rules from scratch.
> * **Acceptance Criteria:** The system successfully mounts the game instance in headless mode, initializes a base Q-table or neural network weights, and logs the first epoch step into the PostgreSQL database in under `t < 3.0s`.

> **US-02: Algorithmic Code Mutation by the LRM Agent**
> * **As an** R&D Engineer,
> * **I want** the Hermes 3 model to analyze the historical context and generate a new mutation of the Genetic Algorithm when fitness metrics plateau for over 50 generations,
> * **So that** the system can overcome local convergence traps.
> * **Acceptance Criteria:** The system passes the historical fitness curve as context to OpenRouter (Hermes 3), receives a syntactically valid code patch, and displays the complete Chain-of-Thought (reasoning tokens) in a dedicated NBP-gold accent panel.

> **US-03: Secure Code Execution via OpenCode Worker**
> * **As a** System Administrator,
> * **I want** the dynamic GA mutation code generated by the LLM to run inside an isolated OpenCode container,
> * **So that** arbitrary or hazardous code cannot compromise the host environment.
> * **Acceptance Criteria:** Any sandbox escape attempt (e.g., unauthorized system calls, host file system operations) triggers an immediate thread termination, logs the exit error code in PostgreSQL, and alerts the orchestrator to request an immediate code correction.

> **US-04: API Resilience via OpenAI Codex Cascade**
> * **As a** DevOps Engineer,
> * **I want** code generation requests to automatically and transparently route to the OpenAI Codex API (via backup ChatGPT keys) if the OpenRouter platform experiences downtime,
> * **So that** long-running evolutionary experiments proceed uninterrupted.
> * **Acceptance Criteria:** The failover mechanism engages silently within the backend layer after two consecutive timeouts, and a critical notification event is permanently written to the PostgreSQL audit log.

### 1.5. Non-Functional Requirements
* **Sandbox Isolation:** The execution lifespan of an OpenCode container for a single genome evaluation is capped at 60 seconds to mitigate infinite loop vulnerabilities.
* **Database Performance:** PostgreSQL ingestion of game states and chromosome parameters is optimized to handle up to 500 records per second using transaction bundling (bulk inserts).
* **Visual Compliance:** The frontend user interface must strictly adhere to the palette of the `nbp.pl` portal, ensuring a WCAG contrast ratio of at least 4.5:1.

### 1.6. Success Criteria and Key Performance Indicators (KPIs)

| KPI ID | Metric Name | Target Value | Measurement Methodology |
| :--- | :--- | :--- | :--- |
| **KPI-01** | Evolutionary Convergence Rate | > 90% of experiments | Successfully clearing Stage 1 of the selected game (Super Mario/Pacman) before reaching the 200th genetic generation. |
| **KPI-02** | OpenCode Worker Efficiency | < 5% syntax rejection rate | Ratio of code compiled and successfully executed within the sandbox relative to the total mutations emitted by the LRM. |
| **KPI-03** | LLM Cascade Stability (Failover Latency) | < 4.0 seconds | Time elapsed to migrate context and receive a structural fix from OpenAI Codex following an OpenRouter node failure. |

## 2. ARD (Architectural Reference Document)

### 2.1. Polyglot Microservices Architecture
The system is designed around a distributed polyglot microservice pattern, optimizing the choice of technology for the respective mathematical and infrastructure tasks:

| Layer / Component | Technology | Architectural Justification & System Role |
| :--- | :--- | :--- |
| **Presentation & Monitoring** | `React.js / TypeScript` | Renders the 2D simulator viewport grid and visualizes the evolutionary family tree of the genetic algorithm. Typography and layouts mirror the **nbp.pl** service. |
| **AI State Management & Orchestration** | `Python / LangGraph / LangChain` | The logical core. `LangGraph` enforces a stateful, cyclic evolutionary loop (*Population -> Evaluate -> Mutate via LRM -> Next Gen*). `LangChain` unifies access across OpenRouter and OpenAI. |
| **Simulator Integration & Structural Analysis** | `Java / LangChain4j / Spring Boot` | Java handles multi-threaded, low-overhead emulator runtimes. `LangChain4j` ensures structural parameter mapping, type-checking, and strong typing boundaries before execution. |
| **BFF & Streaming Gateway** | `Node.js / Express / WebSockets` | Manages asynchronous, low-latency game frame transfers directly from the emulator backends to the React frontend, alongside API gateway auth. |
| **Persistent Data Layer** | `PostgreSQL (v16)` | Relational engine providing strict ACID compliance. It stores complete chromosome arrays, neural weights, Q-tables, LRM session history, and generation telemetry. |

### 2.2. Agent Topology and the Reasoning-Evolution Loop
Decision-making and code transformations operate via a directed cyclic graph managed by the Python layer (`LangGraph`). The pipeline follows this sequence:
* **Step 1 (Evaluation in Java/OpenCode):** The Java microservice executes the 2D game environment. The active control code performs actions within the emulator. Results (score, survival time, win status) are compiled and pushed to PostgreSQL.
* **Step 2 (Stagnation Analysis):** A monitoring agent tracks the fitness function trend line in PostgreSQL. If progress stalls, the Reasoning Node is activated.
* **Step 3 (Mutation Generation via Hermes 3):** Through **OpenRouter**, the **Hermes 3** model is fed the current genetic algorithm code alongside historical telemetry. Leveraging reasoning tokens, the model adjusts the algorithm (e.g., changes mutation rates, alters selection mechanics, redefines reward shaping). If the endpoint fails, a transparent failover routes to **OpenAI Codex**.
* **Step 4 (Verification & Execution):** The mutated code block is handed down to the **OpenCode Worker**. Upon passing syntactic checks, it serves as the algorithmic engine for the next generation of agents (Loop back to Step 1).

### 2.3. Visual Identity Design Standards (NBP.pl Canvas)
The frontend layer intentionally avoids generic modern tech aesthetics, adopting instead an authoritative institutional framework derived from the National Bank of Poland:
* **Base Color Palette:** The dominant background and heading color is a deep corporate navy blue (`#002C5B`), representing stability and authority. Secondary accents, active states, and borders utilize a muted gold (`#B59A57`). Workspace canvasses use a light matte cream hue (`#FCFBFA`).
* **Typography & Grid:** Primary headings employ traditional serif typography (*Georgia* / *Times New Roman*), while statistical data tables use clean sans-serif layouts (*Arial*). Logos are encased within a geometric badge with classic, formal lettering.

## 3. SDLC PLAN (Project Plan and Schedule)

The project will follow an Agile Scrum methodology over a strict 6-month Software Development Lifecycle (SDLC), utilizing 2-week sprints to ensure continuous iterative validation of evolutionary gains.

### 3.1. Phase 1: Initiation, Mathematical Modeling & NBP UX (Months 1-2)
* Formulation of full design tokens and UI mockups in Figma, replicating the conservative color grids and serif typographic hierarchies of NBP.pl.
* Database schema implementation in PostgreSQL (structuring tables for chromosomes, neural matrices, and environment states).
* Setting up the Java runtime to initialize and pipeline *Super Mario* and *Pacman* engine wrappers in headless execution modes.

### 3.2. Phase 2: Core Implementation & Sprint Scheduling (Months 3-4)
Parallel development cycles mapped across functional technology domains:

| Timeline (Sprints) | Technical Domain | Scope of Work & Milestones |
| :--- | :--- | :--- |
| **Sprint 1 - 2** | Base Infrastructure & Node.js Gateway | Setting up automated CI/CD pipelines (GitHub Actions). Deploying the Node.js BFF gateway with WebSocket infrastructure for game-frame streaming. Implementing NBP layout standards in React.js. |
| **Sprint 3 - 4** | AI Core Orchestration (Python / LangGraph) | Developing the stateful evolutionary graph in LangGraph. Wiring OpenRouter connections for Hermes 3. Creating prompt templates for algorithmic mutation and managing reasoning token ingestion. |
| **Sprint 5 - 6** | Java Layer (LangChain4j) & OpenCode | Implementing the Java microservice for genome parameter type verification. Integrating the OpenCode sandbox execution loop. Engineering the seamless failover pipeline to OpenAI Codex. |

### 3.3. Phase 3: Advanced Testing, Optimization & Quality Assurance (Month 5)
* Validating algorithmic resilience against premature convergence and genetic drift anomalies.
* Executing rigorous penetration testing against the OpenCode sandbox to verify isolation boundaries against privilege escalation or malicious container escapes.
* Granting closed beta access to a cohort of 500 engineers and researchers to evaluate LRM reasoning transparency and NBP-style dashboard utility.

### 3.4. Phase 4: Production Deployment & Maintenance (Month 6)
* Containerizing all components via Docker and deploying the topology onto a production-ready Kubernetes cluster.
* Wiring Node.js, Python, and Java logging hooks into Datadog and Firebase Performance to continuously monitor LRM response latencies and API network errors.
* Handover to operations, configuring automated snapshot backup policies for the PostgreSQL instances containing the final, optimized agent genomes.
